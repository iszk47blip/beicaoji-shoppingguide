## Problem Statement

门店运营需要一个一站式仪表盘，覆盖四个维度：收入与订单量、商品动销排行、AI 推荐转化漏斗、顾客画像。当前 staff-admin 只有 CRUD 管理功能，缺少数据洞察视图。

## What Makes This Cool

- **AI 转化漏斗是这个系统最有差异化的报表** — 不是通用的 GMV 看板，而是能看到"AI 推荐了什么 → 顾客买了什么"的闭环
- **时间维度联动** — 切换时间范围，四个维度的图表全部刷新
- **Mock 数据先跑完整版** — 用模拟的 30-90 天数据验证设计，真实数据接入后零改动

## Constraints

- 当前真实数据极少（1 订单、0 对话、0 顾客），需要完整的 mock 数据生成器
- 复用 staff-admin 现有 UI 框架（侧边栏导航、CSS 变量、卡片/表格风格）
- 国内用户访问，CDN 资源需选择国内可达的节点

## Premises

1. 报表模块只读，纯粹聚合展示，新增 `/api/staff/reports` router
2. Mock 数据生成器模拟 30-90 天订单/对话/顾客数据，商品用真实 254 件
3. 复用 staff-admin UI 框架，侧边栏加"报表"tab
4. AI 转化漏斗是核心差异化维度
5. 先完整后简化 — mock 版做全四个维度

## Approaches Considered

### Approach A: 纯 HTML 报表
- Summary: 后端聚合 JSON，前端 CSS div 柱状图 + KPI 卡片 + 表格
- Effort: S | Risk: Low
- Pros: 零依赖、快速交付
- Cons: 无趋势图、交互弱

### Approach B: ECharts 全功能仪表盘 (CHOSEN)
- Summary: CDN 引入 ECharts，折线图/漏斗图/饼图/柱状图 + 时间筛选器
- Effort: L | Risk: Low
- Pros: 专业可视化、四个维度各有匹配图表类型、中文支持成熟
- Cons: CDN 依赖、代码量最大

### Approach C: 嵌入式上下文统计
- Summary: 不新建 dashboard，统计嵌入现有列表页
- Effort: M | Risk: Medium
- Pros: 场景化
- Cons: 缺少整体视图、用户明确要一站式

## Recommended Approach

**Approach B: ECharts 全功能仪表盘**

### 技术选型
- **图表库**: ECharts 5.x，通过 jsDelivr CDN 加载（国内用户可用 bootcdn 备用）
- **时间筛选**: 预设按钮组（今日/7天/30天/90天）+ 自定义日期范围
- **数据层**: 新增 `backend/app/api/reports.py`，所有端点返回预聚合 JSON
- **Mock 数据**: 独立脚本 `backend/mock_data.py`，生成可配置天数的模拟数据

### API 设计

```
GET /api/staff/reports/overview?start=2026-04-22&end=2026-05-22
→ { total_revenue, order_count, avg_order_value, ai_conversion_rate }

GET /api/staff/reports/revenue-trend?start=...&end=...&granularity=day
→ [{ date, revenue, order_count }, ...]

GET /api/staff/reports/product-ranking?start=...&end=...&limit=10
→ [{ sku_id, name, category, times_recommended, times_purchased, revenue }, ...]

GET /api/staff/reports/conversion-funnel?start=...&end=...
→ { conversations, recommendations, add_to_cart, purchases, rates: {...} }

GET /api/staff/reports/category-distribution?start=...&end=...
→ [{ category, revenue, order_count, percentage }, ...]

GET /api/staff/reports/constitution-distribution?start=...&end=...
→ [{ constitution_type, customer_count, avg_order_value }, ...]
```

### Mock 数据生成器设计

```python
# 生成参数
--days 60          # 模拟 60 天数据
--orders-per-day 3 # 日均 3 单
--seed 42          # 可复现的随机

# 生成逻辑
1. 用真实 Product 表（254 件）作为商品池
2. 生成 Customer（100 位），随机分配体质
3. 生成 Conversation（每天 5-10 条），关联顾客
4. 生成 Order（每天 1-5 单），含 items_json + snapshots
5. AI 推荐转化率控制：30% 对话产生推荐，15% 推荐产生加购，50% 加购产生成交
```
